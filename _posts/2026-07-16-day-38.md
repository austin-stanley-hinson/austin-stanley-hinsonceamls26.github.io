what_i_learned: |
  Today was hard, but it was also one of the most important technical days of the
  project. I focused on expanding the structure-based validation part of our
  Alzheimer's drug discovery workflow by running multiple docking tools on the
  final top-12 AChE candidate compounds.

  I first finalized the local AutoDock Vina 1.2.7 results and confirmed the
  summary CSV containing the ranked docking scores. After that, I set up GNINA
  using Docker on my Mac, tested the default CNN scoring mode, and successfully
  generated GNINA docking outputs with affinity, CNN score, and CNN affinity
  values.

  After GNINA, I moved on to rDock as another open-source docking comparison. I
  generated the AChE binding cavity using the donepezil reference ligand, docked
  all 12 compounds, and created the rDock summary CSV. Some rDock outputs produced
  warning flags, so I noted that those results should be treated as supporting
  evidence rather than the main decision source.

  The most challenging part of the day was setting up DOCK6. I had to build DOCK6
  from source inside Docker, fix missing dependencies like zlib, create a Docker
  runner image, prepare receptor and ligand MOL2 files, troubleshoot strict MOL2
  formatting issues, manually define the AChE docking box using the same
  Vina/GNINA grid center, and successfully generate DOCK6 grid files.

  After resolving those setup issues, I ran DOCK6 docking on the full top-12 set
  and generated the DOCK6 summary output.

  By the end of the day, I had results from four docking and scoring approaches:
  AutoDock Vina, GNINA, rDock, and DOCK6. This gave the project a stronger
  structure-based validation layer because the final compounds could now be
  compared across multiple docking methods instead of relying on only one tool.

blockers: |
  The main blocker today was the difficulty of setting up and troubleshooting
  multiple docking tools, especially DOCK6. Building it from source inside Docker,
  fixing missing dependencies, preparing strict MOL2 files, and generating the
  docking grid took a lot of time and debugging.

  Another challenge was that each docking tool produced different types of scores
  and outputs, so I had to be careful not to compare raw scores directly across
  tools.

reflection: |
  Today helped me understand why consensus docking is useful. Since AutoDock Vina,
  GNINA, rDock, and DOCK6 use different scoring approaches, it would not be fair
  to directly compare their raw scores as if they were the same measurement.
  Instead, the better approach is to compare each compound's rank within each
  tool and then calculate a final consensus rank across all tools.

  I also planned the final consensus docking workflow in Colab. The workflow will
  upload each tool's CSV output, display the individual rankings with compound
  names and scores, merge the results by Compound_ID, and calculate a final
  consensus rank based on average ranking across tools.

  Even though today was difficult, it moved the project from single-tool docking
  toward a stronger validation workflow using multiple independent docking
  approaches.
